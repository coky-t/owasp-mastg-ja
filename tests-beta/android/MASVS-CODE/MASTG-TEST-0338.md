## 概要

アプリは機密データをローカルストレージに保存し、後でセキュリティ関連の判断に使用することがあります。そのデータが攻撃者によって改変される可能性があり、アプリが使用前にその完全性と真正性を検証しない場合、アプリは改竄された入力を信頼してしまう可能性があります。

このテストは `SharedPreferences`、ファイル、データベース、その他のアプリが管理するストレージ場所など、ローカルストレージ全般に適用します。

たとえば、具体的には `SharedPreferences` を使用する場合、データはアプリのプライベートサンドボックスに保存され、通常は他のアプリによって変更されることはありません。しかし、ルート化されたデバイス、動的解析時、バックアップ経由、または特権アクセスを取得した後にアプリのデータディレクトリを直接操作するなど、[共有プリファレンス (Shared Preferences)](../../../knowledge/android/MASVS-STORAGE/MASTG-KNOW-0036.md) で説明されているように、ローカル攻撃シナリオでは改竄される可能性が依然としてあります。そのため、アプリはローカルストレージからロードされるセキュリティ関連データを盲目的に信頼すべきではありません。

このテストを実行する際は、ストレージ読み取り API だけでなく、その周辺にある完全性と真正性バリデーションロジックも探します。実装によっては、これは HMAC、MAC 比較、暗号初期化、署名検証、チェックサム、改竄検出を目的としたその他のメカニズムに関連する API やパターンを含むことがあります。

## 手順

1. [Java コードの逆コンパイル (Decompiling Java Code)](../../../techniques/android/MASTG-TECH-0017.md) でアプリをリバースエンジニアします。
2. [Android での静的解析 (Static Analysis on Android)](../../../techniques/android/MASTG-TECH-0014.md) でリバースエンジニアしたアプリに対して静的解析ツールを実行し、ローカルストレージからデータを読み取る API、および可能であれば関連する完全性と真正性のバリデーション API を特定します。

## 結果

出力にはアプリがローカルストレージからデータを読み取るコード箇所を含む可能性があります。ストレージ API と解析ルールによっては、これらのコード箇所には、`SharedPreferences.getString` などの API、ファイル読み取り、データベースクエリ、あるいは近くの HMAC や MAC 関連の演算などの比較と検証のロジックを含むことがあります。

## 評価

アプリが、セキュリティ関連の判断に使用する前に、ローカルストレージからロードしたデータの完全性と真正性を検証しない場合、そのテストケースは不合格です。

これは、たとえば `SharedPreferences.getString` のようなターゲット API の存在が本質的にテストを不合格にするわけではないことを意味します。報告された各コード箇所はリバースエンジニアリングによって慎重に解析し、アプリが適切なバリデーションを実行しているかどうかをチェックする必要があります。

報告されたコード箇所を評価する際には、以下を確認します。

1. ローカルストレージからロードされている値は何か。
2. その値が、認証状態、認可、機能アクセス、設定、信頼性判断など、セキュリティ関連の判断に影響を与える可能性があるかどうか。
3. アプリが、たとえば HMAC、MAC、署名、または同様の検証メカニズムで、使用前に、ロードされた値の完全性と真正性を検証しているかどうか。
4. そのバリデーションは対象とする攻撃モデルに対して有効であるかどうか。
